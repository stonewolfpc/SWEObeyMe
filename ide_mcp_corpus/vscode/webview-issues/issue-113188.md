# GitHub Issue #113188: Switching back and forth in WebView View with retainContextWhenHidden causes ghost renders

**Issue URL:** https://github.com/microsoft/vscode/issues/113188

**Reporter:** agrawal-d  
**Date:** Dec 20, 2020  
**Issue Type:** Bug

## Description

With `retainContextWhenHidden=true`, there appears to be a rendering issue causing ghost/overlapping renders when the webview view changes when it is hidden.

### Steps to Reproduce

1. Have an extension using webview view, and `retainContextWhenHidden=true`. This extension renders something that changes whenever the active editor changes.
2. Navigate to a file that renders something on to the webview view.
3. Navigate to Explorer view, such that the folders list replaces the webview view.
4. Open a file from the folders list, this makes the extension to bring the webview view to focus and render something else.
5. However, the previous render remains (only render, not the HTML/DOM) and overlaps with the new render. Resizing the view fixes this.

### Important Notes

- This is not a DOM/programming error - there is no duplicate HTML being rendered
- The ghost render does not exist in the DOM, but it can still be seen (verified from inspect tool)
- This bug does not happen if the webview view never gets hidden
- This issue does not occur if `retainContextWhenHidden=false`

## Analysis

This issue demonstrates that `retainContextWhenHidden` has rendering bugs in VSCode. The ghost/overlapping renders suggest that VSCode's webview implementation has issues with:
1. Preserving webview state when hidden
2. Properly re-rendering when visibility changes
3. Clearing previous render artifacts

This is related to issue #106693 where webviews refresh to initial state after becoming hidden and visible again.

## Root Cause

VSCode's webview implementation has bugs when using `retainContextWhenHidden`:
- Webview content is not properly preserved during visibility changes
- Previous render artifacts persist (ghost renders)
- Webview may reset to initial state instead of preserving current state

## Implications for SWEObeyMe

The "long list of 152 options" that appears when clicking "Open Full Settings" could be:
1. A ghost render of a previous state
2. The webview being reset to initial state
3. A fallback state when proper state preservation fails

## Recommended Solution

Disable `retainContextWhenHidden` and implement proper state persistence:
1. Use `getState()`/`setState()` for webview state
2. Use extension context globalState for persistent state
3. Restore state manually when webview becomes visible
4. Listen to `onDidChangeViewState` to detect visibility changes
